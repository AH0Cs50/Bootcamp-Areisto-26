# 🎬 Movie API (Node.js - Native HTTP)

A RESTful API for managing movies using **Node.js core modules only (no Express)**.  
Data is stored in JSON files with support for full CRUD operations, search, and partial updates.

## 🧱 Architecture Overview

This project follows a **layered architecture inspired by Domain-Driven Design (DDD)**:

```
Controller → DTO → Mapper → Entity → Repository → JSON File
```
> make router.js follow express framework behavior and middleware to collect body request
---

## 🔹 Layers Explained

### 1. Controller
- Handles HTTP requests/responses
- Extracts params, query, and body
- Delegates logic to lower layers

---

### 2. DTO (Data Transfer Object)
- Validates incoming request data
- Normalizes values (e.g., year format)
- Supports:
  - Create (full payload)
  - Update (partial payload)

---

### 3. Mapper
- Responsible for transforming data between layers:
  - DTO ↔ Entity
  - Entity ↔ Database
- Handles:
  - Field renaming (`rating → vote_average`)
  - Array ↔ string conversion

---

### 4. Entity (Movie.js) ⭐

The **Movie entity represents the core domain model**.

This layer is responsible for:

- Encapsulating **business rules**
- Handling **domain logic**
- Protecting data integrity

#### Examples of what belongs here:
- Calculating derived values (e.g., ROI = revenue - budget)
- Enforcing business constraints
- Domain-specific methods

> This ensures business logic is NOT scattered across controllers or repositories.

---

### 5. Repository
- Handles data persistence
- Reads/writes JSON files
- Provides CRUD operations

---

### 6. Storage (JSON Files)
- `movies-db.json` → stores movies
- `meta.json` → stores last generated ID and other meta data 

## 📌 API Endpoints

### Get all movies
```
GET /movies
```

### Search movies by title
```
GET /movies?search=batman
```

### Limit number of results
```
GET /movies?limit=20
```

---

### Get movie by ID
```
GET /movies/:id
```

---

### Create a new movie
```
POST /movies
```

---

### Update a movie (partial update)
```
PATCH /movies/:id
```

---

### Delete a movie
```
DELETE /movies/:id
```

## 🧠 Assumptions

- `title`, `rating`, and `year` are required to create a movie
- `year` can be:
  - a number (e.g., `2010`)
  - a string (e.g., `"10/10/2010"`)
- Missing date parts in `year` are normalized:
  - `2010` → `00/00/2010`
  - `10/2010` → `00/10/2010`
- Year is stored as a string in format:
```
dd/mm/yyyy
```
- ID generation:
  - A separate JSON file (`meta.json`) stores the last generated ID
  - New ID is generated by incrementing `last_id`
- Default values:
  - string → `""`
  - number → `0`
  - boolean → `false`
- If required fields are missing → movie creation is rejected

## ❌ Error Handling

The API is designed to handle common errors and return **clear, consistent JSON responses**.

All errors follow this structure:

```json
{
  "success": false,
  "message": "Error description"
}
```

---

### 🔹 Types of Errors Handled

#### 1. Invalid Route
- Returned when the requested endpoint does not exist

**Response:**
```json
{
  "success": false,
  "message": "Route not found"
}
```

**Status Code:** `404 Not Found`

---

#### 2. Movie Not Found
- Returned when a movie with the given ID does not exist

**Response:**
```json
{
  "success": false,
  "message": "Movie not found"
}
```

**Status Code:** `404 Not Found`

---

#### 3. Invalid Request Body
- Returned when request validation fails (DTO validation)

Examples:
- Missing required fields (`title`, `rating`, `year`)
- Invalid data types

**Response:**
```json
{
  "success": false,
  "message": "Validation error message"
}
```

**Status Code:** `400 Bad Request`

---

#### 4. File Read/Write Errors
- Returned when there is a problem accessing the JSON file

**Response:**
```json
{
  "success": false,
  "message": "Error reading or writing data"
}
```

**Status Code:** `500 Internal Server Error`

---

#### 5. Unexpected Server Errors
- Catches any unhandled exceptions in the application

**Response:**
```json
{
  "success": false,
  "message": "Internal server error"
}
```

**Status Code:** `500 Internal Server Error`

---

### 💡 Notes

- All errors are handled using a centralized helper (`sendError`)
- Proper HTTP status codes are used for each scenario
- Validation errors come from the DTO layer
- Repository errors (file system) are propagated and handled in controllers
